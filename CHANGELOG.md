# Changelog

All notable changes to Cluster Optimizer will be documented in this file.

## Unreleased

- Added `scripts/node-image-gc-live-runbook.md`, an operator runbook for enabling the node image GC DaemonSet in live mode. It covers image pinning to the deployed CronJob tag, GHCR pull-secret wiring, the live deletion gate, first-run removal caps, rollout/log verification, storage-impact checks, and dry-run/disable/halt rollback commands. Documentation only.
- Added node disk-utilization detection and an opt-in node image garbage collector that together address the "node disk filling up" alert (e.g. DigitalOcean's default 70% droplet disk warning) that the kubelet's own image GC never resolves on managed clusters, because it only triggers at 85% and is not tunable on DOKS. **Detection:** the collector now fetches each node's kubelet `stats/summary` (via a read-only `nodes/proxy` grant added to `manifests/rbac.yaml`) to record root-disk capacity/used and image-filesystem usage, plus the node's `DiskPressure` condition; the new `node-disk-utilization-high` analyzer rule flags nodes at or above 70% (medium) / 85% or under `DiskPressure` (high) and points at the pruner. Detection is read-only and runs in the existing CronJob; a node whose kubelet stats are unreachable is skipped rather than failing the run. **Active cleanup:** a new `cmd/node-image-gc` binary (shipped in the same image) runs as an opt-in DaemonSet (`manifests/daemonset-image-gc.yaml`), one pod per node, talking to the node's containerd socket over the CRI gRPC API to remove images no container references — the stale per-deploy image versions that dominate node disk. It measures disk via `statfs` on a read-only host mount and only prunes above a configurable threshold (`CLUSTER_OPTIMIZER_NODE_GC_DISK_THRESHOLD`, default 65% — below the 70% alert and the 85% kubelet GC). Mirroring every other mutation path, it is **dry-run by default** (logs what it would remove), goes live only with `CLUSTER_OPTIMIZER_NODE_GC_LIVE=true`, honours the shared halt ConfigMap, supports a `CLUSTER_OPTIMIZER_NODE_GC_MAX_REMOVALS` blast-radius cap (largest images first), and relies on the CRI's own "image in use" refusal as a second safety net. The reclaim logic is pure and unit-tested against a fake CRI (dry-run, live, threshold skip, cap, tag-vs-ID in-use matching, per-image error tolerance); the `stats/summary` parser and the disk rule have unit tests too. The DaemonSet uses a minimal namespaced ServiceAccount/Role (`cluster-optimizer-node-gc`) that can only read the halt ConfigMap; it is not privileged (runs as root only to reach the root-owned socket, drops all capabilities, read-only root filesystem). Pins `k8s.io/cri-api@v0.35.6`, `google.golang.org/grpc@v1.72.2`, and stable `google.golang.org/protobuf@v1.36.8`; only the `node-image-gc` binary links gRPC, so the optimizer/CronJob binary stays lean.

- Added a `docker-compose.yml` and `scripts/create-local-table.sh` that stand up a local DynamoDB (via [DynamoDB Local](https://hub.docker.com/r/amazon/dynamodb-local)) so the engine's persistence path (`PutReport`/`PutRemediations`/`PutEngineStatus`) can be exercised end-to-end without touching real AWS — useful for validating, e.g., that a completed-pod GC run writes its `delete_completed_pod` remediation audit event. No application code change was needed: `aws-sdk-go-v2/config` (v1.32) honors `AWS_ENDPOINT_URL_DYNAMODB` natively, so pointing the CLI at the local endpoint is purely a matter of environment variables (documented in the compose file header). The compose service binds host port `8001` to avoid colliding with other local DynamoDB instances, and the table-creation script is idempotent and uses dummy `local`/`local` credentials that DynamoDB Local ignores. Local tooling only — nothing ships in the deployed CronJob image.
- Documented the completed-pod GC opt-in flags (`--gc-namespace`, `--gc-min-age`, `--gc-max-deletions`) in the commented guidance of the public example manifest `manifests/cronjob.yaml`, so operators scoping and bounding blast radius can see the available guardrails inline. Docs-only: the live gate (`CLUSTER_OPTIMIZER_GC_COMPLETED_PODS_LIVE`) stays off by default and the GC block remains commented out, so applying the example never deletes pods until an operator explicitly opts in.
- Renamed the cluster-optimizer `ClusterRole`/`ClusterRoleBinding` from `cluster-optimizer-readonly` to `cluster-optimizer-engine`, since the role is no longer read-only (it grants `update nodes`, `create pods/eviction`, `delete pods`, and workload patches for the opt-in remediation paths). Added `scripts/apply-rbac.sh`, an idempotent helper that applies `manifests/rbac.yaml` and removes the pre-rename objects so a renamed apply does not orphan them — this is the script to run when `scripts/verify-deployment.sh` reports RBAC drift (e.g. `delete pods (completed-pod GC): no`). The Deploy Kubernetes workflow now calls this script instead of a bare `kubectl apply`, and the README/verify-script guidance points at it. Existing clusters pick up the rename (and the `delete pods` grant) on the next deploy or by running `scripts/apply-rbac.sh` directly.
- Added a `podgc` package and `--gc-completed-pods` flag that cleans up completed (phase `Succeeded`/`Failed`) pods left behind by Jobs and CronJobs — the kind that pile up from recurring jobs (e.g. `ecr-pull-secret-refresh-*`) and clutter `kubectl get pods`, count against the per-node pod cap, and hold IP allocations. Like the nudger, it is dry-run by default and shares the halt switch; live deletion requires `CLUSTER_OPTIMIZER_GC_COMPLETED_PODS_LIVE=true`. Optional guardrails: `--gc-namespace` (scope to one namespace), `--gc-min-age` (only delete pods that finished at least N ago, e.g. `1h`), and `--gc-max-deletions` (cap deletions per run, oldest first). Each run records a `delete_completed_pod` remediation audit event (new `deleted`/`deletion_errors` fields) and the engine status sentinel gains `gc_enabled`/`gc_live`. Granted the `delete` verb on `pods` in `manifests/rbac.yaml` (required only for live GC) and added the GC pass (dry-run) to the full-engine `examples/cronjob-dynamodb.yaml`. Extended `scripts/verify-deployment.sh` with a soft RBAC drift check (`VERIFY_RBAC`, default `auto`) that uses a SubjectAccessReview to confirm the deployed ServiceAccount actually has the pod verbs the engine needs — including the new `delete pods` — so an out-of-date role is caught before a live remediation path silently 403s; it skips cleanly when the kubeconfig can't impersonate. Added unit tests covering dry-run, live deletion, halt (active + fail-closed), namespace scoping, phase selection, min-age, the deletion cap, terminating-pod skip, and the `podGCEvent` audit mapping (incl. the all-namespaces empty-namespace contract).

- Fixed the resource-usage analyzer (`analyzeUsage`) so the `memory-request-over-provisioned`, `memory-request-below-usage`, and `cpu-request-over-provisioned` rules compare **per-replica** request and usage instead of fleet totals. The collector sums requests and usage across every replica (`aggregatePods`), so a multi-replica workload whose per-pod request was already reasonable was flagged as over-provisioned purely because of its replica count — e.g. a Deployment with 2 replicas at 128Mi each read as a 256Mi request "less than half" of 47Mi observed. The remediation target derived from that evidence (and the patch the planner applies) is per container, so the recommendation told operators to "set memory request to 128Mi" on a container already at 128Mi: a no-op that re-fired on every run and accumulated hundreds of occurrences that could never resolve. The rule now divides by replica count (matching the per-replica math already used by `analyzeRuntimeModernization`, `analyzeHPASensitivity`, and `plan.planRequestTrim`), skips workloads with no ready replicas, and emits per-pod values in the evidence so the UI's `remediationTargetsFromEvidence` derivation stays consistent. Genuinely over-provisioned workloads still fire and now naturally stop once their per-pod request is trimmed. Added regression tests covering the reported echothread-api/tovinio-backend scenarios, the per-replica evidence contract, the CPU path, and the UI target derivation.

- Replaced the activity feed's generic `why?` link on skipped events with action-led links that only appear when the user can actually do something about the skip.
